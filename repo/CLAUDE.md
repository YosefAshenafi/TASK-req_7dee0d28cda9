# FulfillOps Rewards & Compliance Console

## Project Overview

Offline-first web application for end-to-end reward fulfillment operations. Manages reward tiers, fulfillment lifecycle (Draft → Ready to Ship → Shipped/Voucher Issued → Delivered → Completed, with On Hold and Canceled), inventory with concurrency safety, purchase limits, SLA enforcement, messaging, audit trails, and backup/DR.

## Tech Stack

- **Backend**: Go 1.22+ with Gin framework
- **Frontend**: Templ templates (server-rendered), HTML, CSS, JavaScript
- **Database**: PostgreSQL 16
- **Infrastructure**: Docker / Docker Compose for all operations

## Project Structure

```
cmd/server/main.go          # Application entry point
internal/
  config/                   # Environment configuration
  domain/                   # Models, enums, errors (no external deps)
  repository/               # PostgreSQL data access, transactions
  service/                  # Business logic (fulfillment, inventory, SLA, messaging, encryption, audit)
  handler/                  # Gin HTTP handlers + router
  middleware/               # Auth, logging, error handling, request tracing
  job/                      # Scheduled job definitions + scheduler
  view/                     # Templ templates organized by page/feature
  util/                     # Validation, PII masking, CSV helpers
migrations/                 # SQL migrations (golang-migrate format: NNN_name.up/down.sql)
static/                     # CSS, JS, images
docker/                     # Dockerfile + docker-compose.yml
tests/integration/          # End-to-end integration tests
```

## Build & Run

All operations use Docker:

```bash
make up              # Build and start app + PostgreSQL
make down            # Stop all containers
make build           # Build Go binary in Docker
make migrate-up      # Run database migrations
make migrate-down    # Rollback last migration
make templ-generate  # Generate Go code from .templ files
make generate-key    # Create dev encryption key
make test            # Run all tests in Docker
make test-repo       # Repository tests only
make test-service    # Service tests only
make test-handler    # Handler tests only
make test-jobs       # Job tests only
make smoke           # Full integration: build → test → teardown
make lint            # Run golangci-lint
```

## Key Design Patterns

- **Transactions**: Service layer owns tx lifecycle via `TxManager.WithTx`; repos accept tx or pool
- **Concurrency**: `SELECT ... FOR UPDATE` row locks + optimistic versioning (version integer on mutable rows)
- **Encryption**: AES-256-GCM at service boundary; repos store raw `[]byte`
- **PII masking**: Handlers mask decrypted values before response; default masked, explicit permission for sensitive exports
- **State machine**: `AllowedTransitions` map in `domain/enums.go` is single source of truth
- **Audit**: Append-only `audit_logs` table; no UPDATE/DELETE grant for app role
- **Soft-delete**: `deleted_at`/`deleted_by` columns; 30-day recovery window; nightly hard-delete purge

## Testing

- Integration tests run against real PostgreSQL in Docker (no mocks for data layer)
- Repository tests: `internal/repository/*_test.go`
- Service tests: `internal/service/*_test.go`
- Handler tests: `internal/handler/*_test.go` using `httptest`
- Integration tests: `tests/integration/` for full end-to-end flows

## Coding Conventions

- Use `internal/` for all non-exported packages
- Error handling: return domain errors (`ErrNotFound`, `ErrConflict`, etc.); middleware maps to HTTP status
- UUIDs for all primary keys (generated by PostgreSQL `gen_random_uuid()`)
- JSON tags on all structs; `db` tags for database mapping
- Templ components render via `component.Render(ctx, c.Writer)`
- Migrations are sequential numbered: `001_name.up.sql` / `001_name.down.sql`
